# next-stack: Starter Template Design

A reusable starter template for full-stack Next.js apps, extracted from the `example/save-my-stuff` project. Ships a working auth flow, database setup, and deployment pipeline with zero domain-specific code.

## Stack

| Layer | Technology |
|-------|-----------|
| Framework | Next.js 16 (App Router) |
| Runtime | Bun |
| Language | TypeScript (strict) |
| Styling | Tailwind CSS v4 |
| Auth | Better Auth (email/password, Drizzle adapter) |
| Database | PostgreSQL -- Neon serverless (prod), postgres-js (local) |
| ORM | Drizzle ORM (schema push workflow) |
| Deployment | Vercel |
| Local dev DB | Docker Compose (Postgres 16) |
| Code quality | ESLint, Prettier, pre-commit hooks |

## Key Design Decisions

### Dual database client

`lib/db/client.ts` checks `VERCEL=1` at runtime to switch between `@neondatabase/serverless` (Neon HTTP adapter for production) and `postgres-js` (local dev). This means the same codebase works locally against Docker Postgres and in production against Neon without config changes.

### Auth schema is code-generated

Better Auth owns its table definitions. Running `bun run auth:generate` invokes `better-auth/cli generate` to produce `auth-schema.ts` (user, session, account, verification tables). This runs before every build and migration so the schema stays in sync with the auth config. App-specific tables go in `schema.ts` and import from `auth-schema.ts`.

### Schema push over migrations

Uses `drizzle-kit push` rather than the generate-then-migrate workflow. This keeps the dev loop fast and avoids migration file churn during early development. The `push-schema.ts` script wraps this for non-interactive CI use (`yes | bunx drizzle-kit push --force`).

### Vercel-aware URL helpers

`lib/env.ts` resolves the app's base URL with a priority chain: `VERCEL_BRANCH_URL` > `VERCEL_URL` > `NEXT_PUBLIC_SITE_URL` > `localhost:3000`. This means Better Auth works correctly across Vercel preview deployments without manually setting URL env vars per branch.

### Makefile as workflow entrypoint

All common operations go through the Makefile: `make setup` (start DB, push schema), `make dev`, `make build`, `make db-push`, etc. The Makefile exports `POSTGRES_URL` for local dev and handles the `cd src` dance so commands are run from the right directory.

## What was removed from the example

The `example/save-my-stuff` project is a receipt extraction app. The following domain-specific pieces are stripped from the template:

- Receipt schema, API routes, components, hooks, and types
- Claude vision extraction logic (`@anthropic-ai/sdk`)
- S3 blob storage (`@aws-sdk/client-s3`, `@aws-sdk/s3-request-presigner`)
- Image processing (`sharp`)
- All receipt-related docs

## Environment Variables

Required:

| Variable | Where | Purpose |
|----------|-------|---------|
| `POSTGRES_URL` | Local + Vercel | Database connection string |
| `BETTER_AUTH_SECRET` | Local + Vercel | Auth session signing key |

Auto-set by Vercel (no manual config needed):

| Variable | Purpose |
|----------|---------|
| `VERCEL` | Triggers Neon adapter in db client |
| `VERCEL_URL` | Used for auth base URL resolution |
| `VERCEL_BRANCH_URL` | Used for preview deployment URLs |

Optional overrides:

| Variable | Purpose |
|----------|---------|
| `NEXT_PUBLIC_SITE_URL` | Custom production domain |
| `BETTER_AUTH_URL` | Override Better Auth base URL |

## Getting Started (target workflow)

```bash
# 1. Clone and install
bun install

# 2. Start local Postgres and push schema
make setup

# 3. Start dev server
make dev
```
